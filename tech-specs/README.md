# tech-spec presentations — one site

This directory is **one Vercel project that hosts every tech-spec presentation
in this repo**. Each deck is a standalone Vite + React presentation under
`<spec>/presentation/` (generated by the `/presentation` skill); this folder
adds an index ("the gallery") and the build glue that assembles them all into a
single static site.

```
tech-specs/
├── vercel.json          deploy config (Vercel root directory = this folder)
├── build.mjs            orchestrator: builds the gallery + every presentation
├── package.json         scripts: dev / build / preview
├── _gallery/            the index site (Vite + React, the shared design system)
│   └── src/content/presentations.ts   <- THE ONE FILE YOU FILL IN
├── <spec>/presentation/ one standalone deck per spec (slug = the <spec> dir name)
└── dist/                build output Vercel serves (gitignored)
```

Built output:

```
dist/                 the gallery (index of every deck)
dist/<slug>/          one presentation per spec, served at /<slug>/
```

## Add a presentation

Run the skill — it scaffolds the deck and registers it here automatically:

```
/presentation <repo>/tech-specs/<spec-dir>
```

To add one by hand, drop a deck at `<spec-dir>/presentation/` and add an entry
to `_gallery/src/content/presentations.ts`. **Its `slug` must equal the
`<spec-dir>` name** — that string is the URL (`/<slug>/`) and the folder the
build copies into. Same string in both places or the card 404s.

## Develop & preview

```bash
pnpm dev        # the gallery alone, http://localhost:5173 (cards link out)
pnpm build      # assemble the whole site into dist/
pnpm preview    # build, then serve the full site at http://localhost:4173
```

`pnpm build` installs and builds the gallery and every presentation, so it is
the source of truth for what ships. Use `node build.mjs --skip-install` for a
faster rebuild when dependencies are already installed, or
`node build.mjs --only=<slug>` to rebuild the gallery plus a single deck.

## Deploy (Vercel)

This folder is deploy-ready. Create one Vercel project per repo:

- **Root Directory:** `tech-specs` (this folder)
- **Framework Preset:** Other (or leave it — `vercel.json` sets it)
- Build command, output dir, and install are all set in `vercel.json`.

```bash
# from this folder, once you have the Vercel CLI + are logged in:
vercel link        # link/create the project (Root Directory = .)
vercel --prod      # deploy
```

`dist/` is fully static and uses relative asset paths, so it also works on any
other CDN, GitHub Pages, S3, or opened straight from disk.
