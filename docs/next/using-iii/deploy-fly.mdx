This guide deploys the iii engine and your workers to [Fly.io](https://fly.io). The
approach is the same one used everywhere else (see
[Self-hosted deployment](./deployment)): ship a **clean engine image** and let your
`config.yaml` decide what runs. The base image carries **no** workers. You add
capabilities by declaring them, and the engine provisions them. One image stays
reusable across every app.

Fly.io differs from a generic Docker host in four ways that shape the rest of this
guide:

- **Every app is a Firecracker micro-VM** ([Fly Machines](https://fly.io/docs/machines/)).
  This is the natural home for the iii model: the engine runs on one Machine and each
  worker runs on **its own Machine** (its own micro-VM), wired together over the private
  network into one system (see
  [The Fly architecture for iii](#the-fly-architecture-for-iii)). The same isolation is
  also a limit: a Machine does **not** expose `/dev/kvm`, so the engine cannot boot its
  *own* libkrun micro-VM workers inside it (covered in
  [What cannot run on Fly](#what-cannot-run-on-fly)).
- **The private network is IPv6-only** ([6PN](https://fly.io/docs/networking/private-networking/)).
  Apps in one organization reach each other at `<app>.internal`, so the engine must
  bind `[::]` and workers dial it by app name.
- **Durable state is stored on a Fly volume**, an NVMe slice pinned to one Machine in one
  region. It is not network storage and is not replicated.
- **Fly Proxy is your reverse proxy.** It terminates TLS at the edge and forwards a
  public port to your app, so you do not run Caddy or nginx yourself.

## Pre-built images on Docker Hub

iii publishes the engine to [`iiidev/iii`](https://hub.docker.com/r/iiidev/iii). It is
multi-arch (`linux/amd64` and `linux/arm64`), built on
`gcr.io/distroless/cc-debian12:nonroot`, runs as UID `65532`, and has entrypoint
`/app/iii` with default command `--config /app/config.yaml`. There is no shell or
package manager in it.

You can deploy that image directly **only if your engine is stateless or uses external
adapters** (no Fly volume). The reason is ownership: a freshly created Fly volume is
mounted `root`-owned, and the distroless image runs as a non-root user that cannot
write it. For any deploy that persists to a volume, or that runs registry/binary
workers, build a small base from the install script instead, exactly as on a bare host:

```dockerfile
# Dockerfile: clean iii base. Contains NO workers. Runs as root so it can
# write a Fly volume and launch binary registry workers.
FROM debian:bookworm-slim

RUN apt-get update \
    && apt-get install -y --no-install-recommends curl ca-certificates jq libssl3 libcap2 libcap-ng0 \
    && rm -rf /var/lib/apt/lists/*

# Installs both `iii` (engine) and `iii-worker` (the add-on worker daemon).
RUN curl -fsSL https://install.iii.dev/iii/main/install.sh | sh
ENV PATH="/root/.local/bin:${PATH}"

WORKDIR /app
COPY config.yaml /app/config.yaml

# 49134 worker WS, 3111 REST, 3112 stream. Fly publishes only what fly.toml exposes.
EXPOSE 49134 3111 3112
CMD ["iii", "--config", "/app/config.yaml"]
```

<Note>
  Fly volumes are **not mounted at build time**, and a Fly app has no Compose-style
  file mount for `config.yaml`. So `config.yaml` is **baked into the image** (the
  `COPY` above), not mounted. Rebuild and redeploy to change it.
</Note>

## A Fly-ready config file

The only file you customize is `config.yaml`. Two things make it work on Fly: **bind
`[::]`** (so the engine is reachable over the IPv6 6PN and through Fly Proxy) and **put
all durable state under `/data`** (the Fly volume mount):

```yaml
workers:
  # First manager entry is the worker WS port. Bind [::] so worker apps reach it
  # over Fly's IPv6 private network (and so Fly Proxy can reach the public surface).
  - name: iii-worker-manager
    config: { host: "[::]", port: 49134 }

  - name: iii-http
    config:
      host: "[::]"
      port: 3111
      cors:
        allowed_origins: ["*"]   # tighten to your domains in production
  - name: iii-stream
    config:
      host: "[::]"
      port: 3112
      adapter: { name: kv, config: { store_method: file_based, file_path: /data/stream_store } }
  - name: iii-state
    config:
      adapter: { name: kv, config: { store_method: file_based, file_path: /data/state_store.db } }
  - name: iii-queue
    config:
      adapter: { name: builtin, config: { store_method: file_based, file_path: /data/queue_store } }
  - name: configuration
    config:
      adapter: { name: fs, config: { directory: /data/configuration } }
```

<Note>
  Binding `[::]` listens on the Machine's 6PN address, which is what other apps reach
  through `<app>.internal`, and is also reachable by Fly Proxy for the public surface.
  Do not bind `127.0.0.1`: a worker app dialing `iii-engine.internal` would never
  connect. See
  [Listening on 6PN addresses](https://fly.io/docs/networking/private-networking/#listening-for-connections-on-6pn-addresses).
</Note>

## Launch the engine app

Create the app and a `fly.toml` with `fly launch --no-deploy` (it detects the
Dockerfile), or write `fly.toml` by hand. A stateful single-instance engine looks like
this:

```toml
app = "iii-engine"
primary_region = "iad"

[build]
  dockerfile = "Dockerfile"     # the debian base above + your config.yaml

[env]
  III_EXECUTION_CONTEXT = "fly"
  RUST_LOG = "info"

# Durable state. The volume is mounted at /data; config.yaml file_paths live under it.
[mounts]
  source = "iii_data"
  destination = "/data"

# Keep the single stateful engine alive. The default policy is "on-failure", which
# does NOT restart on a clean exit (code 0) -- and the engine exits cleanly when it
# reloads config.yaml at runtime, which would otherwise leave the Machine stopped.
[[restart]]
  policy = "always"

# Public REST only. 49134 (worker WS) and 3112 (stream) stay private on the 6PN.
[[services]]
  internal_port = 3111
  protocol = "tcp"
  # Stateful single-instance engine: never let Fly Proxy stop it. Autostop would
  # drop worker connections, lose in-memory state, and miss cron fires.
  auto_stop_machines = "off"
  auto_start_machines = false
  min_machines_running = 1

  [[services.ports]]
    port = 443
    handlers = ["tls", "http"]
  [[services.ports]]
    port = 80
    handlers = ["http"]
    force_https = true

  # The engine has no /health route and the image has no shell; use a TCP check.
  [[services.tcp_checks]]
    interval = "15s"
    timeout = "2s"
    grace_period = "10s"
```

Create the volume in the same region, then deploy:

```bash
fly volumes create iii_data --region iad --size 1
fly deploy
```

<Warning>
  A Fly volume is tied to **one Machine in one region** and is **not replicated**. With
  `file_based` adapters you therefore run a **single** engine Machine. Do not
  `fly scale count 2` a file-backed engine: the second Machine needs its own volume and
  will not see the first's data. To run multiple engine Machines, move state to external
  adapters first (see [Scale out](#scale-out-redis-rabbitmq-and-postgres)).
</Warning>

## The Fly architecture for iii

An iii deployment is the engine plus the workers that connect to it. There are two ways to
place a worker on Fly, and a deployment can mix them:

- **In the engine Machine** -- declare the worker in `config.yaml` and the engine runs it
  as a child process. It shares the engine's CPU, memory, and `/data` volume. This covers
  any worker that runs as a process (see
  [What cannot run on Fly](#what-cannot-run-on-fly) for the exception).
- **As its own Fly app** -- the worker runs on a separate Machine with its own CPU,
  memory, and root filesystem, scales independently, and can use a different language
  runtime than the engine image. It connects to the engine over the 6PN (see
  [Connect workers over the private network](#connect-workers-over-the-private-network)).

Every Fly Machine is a Firecracker micro-VM, so a worker deployed as its own Fly app runs
in its own micro-VM. The engine cannot boot libkrun micro-VM workers itself on Fly,
because Machines do not expose `/dev/kvm` (see
[What cannot run on Fly](#what-cannot-run-on-fly)). Which split to use, and how many worker
apps to run, depends on your isolation, scaling, and runtime needs.

## Connect workers over the private network

A typical app is the engine plus one or more worker apps that connect to it over the
6PN. Each worker is its **own Fly app** (its own image and language runtime) because the
engine image carries no runtime:

| App | What it is | Key setting |
| --- | --- | --- |
| `iii-engine` | the base image above + your `config.yaml` | volume at `/data`; public port `443 -> 3111` |
| `api-worker` (and friends) | your Node/Python/Rust worker code | `III_URL=ws://iii-engine.internal:49134`; **no public IP** |

Worker apps stay private and reach the engine at `iii-engine.internal`, Fly's internal
DNS name for the engine app over the 6PN:

```bash
# In the worker app
fly secrets set III_URL=ws://iii-engine.internal:49134
# Release any public IPs so the worker is private-only
fly ips list
fly ips release <addr>
```

```javascript
import { registerWorker } from "iii-sdk";
const worker = registerWorker(process.env.III_URL); // ws://iii-engine.internal:49134
// worker.registerFunction(...) / worker.registerTrigger(...)
```

<Note>
  `<app>.internal` resolves to the 6PN addresses of **started** Machines only. If the
  engine is autostopped, the name will not resolve. This is another reason to keep
  `auto_stop_machines = "off"` on the engine. If a worker needs Fly Proxy features
  (load balancing, autostart) on the private side, give the engine a
  [Flycast](https://fly.io/docs/networking/flycast/) address and dial `*.flycast`
  instead of `*.internal`.
</Note>

## Add workers by declaring them

Each entry in `config.yaml` is a worker. That is all there is in iii: everything is a
worker, and you run one by **declaring it**. If a declared worker is not present
locally, the engine fetches it from the registry on boot. For example, add a
SQLite-backed database with nothing extra in the image:

```yaml
  - name: database
    config:
      databases:
        primary:
          url: sqlite:/data/iii.db        # on the Fly volume
```

The engine logs `Worker 'database' not found locally, checking registry...`, then
registers `database::query`, `database::execute`, and the rest as a plain child
process. This is why the base is built from the install script (it ships `iii-worker`
and `libcap-ng0`): registry workers that deploy as **binaries** run as ordinary child
processes and work on Fly. Supply their secrets from `fly secrets` (below).

<Note>
  Auto-provisioning downloads the worker on first boot, which adds cold-start time and
  needs registry access at runtime. For reproducible, fast starts, **pin it into the
  image**: add `RUN iii worker add <name>` to your Dockerfile so the build is your
  version lock.
</Note>

## Key workers and their adapters

The engine is stateless; state is held in workers, and each stateful worker picks a
storage **adapter** in `config.yaml`. The defaults are in-memory, so choosing durable
adapters is the main production step. The full table is in
[Configure workers and adapters](./deployment#configure-workers-and-adapters); the
short version for Fly:

| Worker | Default | On Fly, use |
| --- | --- | --- |
| `iii-state` | `kv` in-memory | `file_based` on `/data`, or `redis` |
| `iii-stream` | `kv` in-memory | `file_based` on `/data`, or `redis` |
| `iii-queue` | `builtin` in-memory | `file_based` on `/data`, or `rabbitmq` |
| `iii-cron` | `kv` in-memory | `file_based` on `/data` |
| `configuration` | `fs` at `./data/configuration` | `fs` at `/data/configuration` |
| `iii-observability` | `memory` | `otlp` to a collector app on the 6PN |

## From in-memory to file-based

The defaults are in-memory, tuned for local development. When a `kv`-backed adapter runs
in-memory the engine logs `DO NOT USE IN_MEMORY STORE_METHOD IN PRODUCTION - DATA WILL
BE LOST ON SHUTDOWN` on boot. Treat that as a release blocker. Move each stateful worker
to `file_based` and point its `file_path` under `/data` (the volume mount), as shown in
[A Fly-ready config file](#a-fly-ready-config-file). The mechanics and the
`configuration`-worker seeding caveat are covered in
[Make storage durable with file_based](./deployment#make-storage-durable-with-file_based).

Two Fly-specific points:

- **Ownership.** A non-root image cannot write a fresh Fly volume. The debian base in
  this guide runs as root, so `/data` is writable. If you build a non-root image, add a
  root entrypoint that `chown`s the mount before dropping privileges.
- **One volume, one Machine.** `file_based` keeps you on a single engine Machine. The
  volume survives deploys and restarts (the root filesystem does not), but it is one
  copy on one host. Rely on [volume snapshots](https://fly.io/docs/volumes/snapshots/)
  plus your own backups, and move to external adapters before you need a second Machine.

## Scale out: Redis, RabbitMQ, and Postgres

When you outgrow single-instance file storage, swap the `file_based`/`builtin` adapters
for external backends. This is what lets you run more than one engine Machine, since the
state is no longer stored on a per-Machine volume. The adapter config is identical to
self-hosting (see
[Scale out with Redis and RabbitMQ](./deployment#scale-out-with-redis-and-rabbitmq));
only the **hostnames and credentials** are Fly-specific:

- **Redis**: [Upstash for Redis](https://fly.io/docs/upstash/redis/) (`fly redis create`)
  gives a private 6PN endpoint, or run your own Redis app. Point the adapter at it and
  supply the URL through `fly secrets`.
- **RabbitMQ**: run it as its own Fly app with a volume; reach it at
  `rabbitmq.internal:5672`.
- **Postgres / MySQL** (for the `database` worker): use
  [Fly Managed Postgres](https://fly.io/docs/mpg/) and set the connection string as a
  secret.

```yaml
  - name: iii-state
    config:
      adapter: { name: redis, config: { url: "${REDIS_URL}" } }
  - name: database
    config:
      databases:
        primary: { url: "${DATABASE_URL}" }
```

```bash
fly secrets set REDIS_URL=redis://... DATABASE_URL=postgres://...
```

For object storage, use the `storage` worker's **remote** providers (`s3`, `gcs`, `r2`,
or [Tigris](https://fly.io/docs/tigris/) which is S3-compatible). They need no local
disk; supply credentials from `fly secrets`.

## Rust: building binaries

You only build a binary when you ship a **custom engine** (a patched build, or your own
fork). Otherwise use the pre-built image. Build from the repository root:

```bash
cargo build --release          # produces target/release/iii
```

On Fly there are two ways to get that binary onto a Machine:

- **Let Fly build it.** Use a multi-stage Dockerfile so Fly's remote builder compiles
  for the Machine's architecture. This is the reproducible default:

```dockerfile
FROM rust:1-bookworm AS build
WORKDIR /src
COPY . .
RUN cargo build --release      # -> /src/target/release/iii

FROM debian:bookworm-slim
RUN apt-get update \
    && apt-get install -y --no-install-recommends ca-certificates libssl3 libcap2 libcap-ng0 \
    && rm -rf /var/lib/apt/lists/*
COPY --from=build /src/target/release/iii /app/iii
COPY config.yaml /app/config.yaml
WORKDIR /app
EXPOSE 49134 3111 3112
CMD ["/app/iii", "--config", "/app/config.yaml"]
```

- **Build locally and COPY.** If you compile on your machine, the **architecture and
  libc must match the Fly Machine**. Fly Machines are glibc Linux; a musl binary will
  not run on the debian base, and an `arm64` binary will not run on an `amd64` Machine
  (Fly defaults to `amd64`; pick the matching `cargo` target, e.g.
  `--target x86_64-unknown-linux-gnu`). You can also drop your binary onto the official
  base image with `COPY target/release/iii /app/iii`; see
  [Build from source](./deployment#build-from-source).

## Adjust the worker manifest for production

A worker's `iii.worker.yaml` describes how it is provisioned and started. The
development defaults favor fast iteration; production wants the opposite. Adjust the
manifest before you ship (full field reference:
[Adjust the worker manifest for production](./deployment#adjust-the-worker-manifest-for-production)):

- `scripts.start`: drop watch mode (`node dist/index.js`, not `npx tsx watch ...`) so a
  crash is a real signal.
- `scripts.install`: install from a lockfile without dev deps (`npm ci --omit=dev`).
- `env`: inject production config; keep secrets out of the file and supply them from the
  environment (`fly secrets`). `III_URL` / `III_ENGINE_URL` are ignored; the engine sets
  the connection URL.
- `runtime.base_image`: pin a specific base image for reproducible builds.

<Note>
  The manifest's `resources` (`cpus`, `memory`) sizes an **engine-managed micro-VM**
  sandbox, which does not run on Fly (no `/dev/kvm`). For a worker that runs as its own
  Fly app, sizing is the Fly Machine size: set it in `fly.toml` (`[[vm]]`) or with
  `fly scale vm`/`fly scale memory` on that worker app, not in `iii.worker.yaml`.
</Note>

## Hardening and security

- **Bind for the 6PN, expose narrowly.** The engine binds `[::]` so the private surfaces
  are reachable inside the org, but only the port you list in `fly.toml` gets a public
  address. Keep `49134` (worker WS) and `3112` (stream) **off** the public services
  list, and run `fly ips release` on worker apps so they have no public IP. See Fly's
  [going-to-production checklist](https://fly.io/docs/apps/going-to-production/).
- **Secrets in the environment.** Set every credential with `fly secrets set`; Fly
  injects them as environment variables, so `${VAR}` / `${VAR:-default}` placeholders in
  `config.yaml` resolve at boot. Never bake secrets into the image or git. Setting a
  secret triggers a redeploy so the engine picks it up.
- **Restrict CORS.** The development default allows every origin. Narrow
  `iii-http.cors.allowed_origins` to the domains that call your API (see
  [Restrict CORS](./deployment#restrict-cors)).
- **Org-level access.** Use Fly [access tokens](https://fly.io/docs/security/tokens/) with
  least privilege and isolate staging from production with separate organizations.

## RBAC for external workers

The main worker port (`49134`) is trusted, internal traffic with no access control. Two
ways to let a worker connect from **outside** the Fly organization:

- **WireGuard into the 6PN** (operator access, dev laptops): `fly wireguard create`
  gives you a tunnel; once up, dial `ws://iii-engine.internal:49134` as if you were
  inside. Good for trusted operators, not for untrusted third parties.
- **A public RBAC listener** (untrusted callers): run a second `iii-worker-manager` on
  `49135` with role-based access control and expose **only** that port through Fly Proxy.

```yaml
  - name: iii-worker-manager
    config: { host: "[::]", port: 49134 }      # trusted, private on the 6PN

  - name: iii-worker-manager
    config:
      host: "[::]"
      port: 49135                                # public RBAC listener
      rbac:
        auth_function_id: my-project::auth-function
        expose_functions:
          - match("api::*")
```

Expose the RBAC port with another service block, terminated by Fly Proxy with a TLS
handler (WebSocket upgrades pass through the proxy):

```toml
[[services]]
  internal_port = 49135
  protocol = "tcp"
  auto_stop_machines = "off"
  [[services.ports]]
    port = 49135
    handlers = ["tls"]
  [[services.tcp_checks]]
    interval = "15s"
    grace_period = "10s"
```

How the listener gates access (allowlist via `expose_functions`, per-connection
`auth_function_id`, optional `middleware_function_id`) is detailed in
[RBAC](./deployment#rbac). Keep `49134` and `3112` private; channels are served through
the RBAC listener at `/ws/channels/{id}` on the same port.

## Reverse proxy and TLS

On Fly you do **not** run your own reverse proxy. Fly Proxy terminates TLS at the edge
and forwards each public port to your app, which is why the engine surfaces here are
published through `fly.toml` service ports rather than an nginx or Caddy config (contrast
the bare-host [Reverse proxies](./deployment#reverse-proxies) section):

- **REST API**: the `[[services]]` block on `internal_port = 3111` with `tls`/`http`
  handlers on `443`. Fly issues and renews the certificate for `iii-engine.fly.dev`; a
  [custom domain](https://fly.io/docs/networking/custom-domain/) works the same way with
  a `fly certs` entry.
- **WebSocket surfaces** (stream `3112`, RBAC `49135`): Fly Proxy passes WebSocket
  upgrades through natively. Expose each on its own service port; each WS surface is
  served at the **root** of its port, so no path rewriting is needed.
- **Keep the rest private.** Do not add public ports for `49134` or `3112` unless an
  external client truly needs them; in-org workers use the 6PN.

## Health checks

The engine has no HTTP health endpoint and the image has no shell, so in-container
`HEALTHCHECK` probes do not work. Use Fly's external checks instead:

- A `[[services.tcp_checks]]` against the REST port (`3111`) is a reliable liveness
  signal (shown in [Launch the engine app](#launch-the-engine-app)).
- For a readiness signal that reflects worker availability, register an HTTP-triggered
  function that returns `200` and add an `[[services.http_checks]]` that probes that
  route.

## Observability

Set `iii-observability` to the `otlp` exporter and point it at an OpenTelemetry collector
running as its own Fly app on the 6PN (`http://otel-collector.internal:4318`). The
protocol/port pairing (traces and metrics over gRPC `4317`, logs over HTTP `4318`) is
the common pitfall and is covered in [Observability](./deployment#observability). There
is no Prometheus scrape endpoint on `9464`; Fly's
[managed metrics](https://fly.io/docs/monitoring/metrics/) cover Machine-level signals.

## What cannot run on Fly

Fly Machines are Firecracker micro-VMs, and Firecracker has no nested virtualization:
`/dev/kvm` is **not** present inside a Machine (`ls /dev/kvm` -> no such file). So any
iii worker that the engine boots as its **own** libkrun guest cannot run there:

- `iii-sandbox` and any OCI/image (engine-managed) worker. They boot guests via libkrun.
  See [Engine-managed workers (micro-VMs)](./deployment#engine-managed-workers-micro-vms).
- Bundle workers dispatched through the same libkrun rails.

<Warning>
  This is not a soft failure. Declaring an engine-managed micro-VM worker (for example
  `iii worker add iii-sandbox`) makes the engine try to start a guest with no `/dev/kvm`;
  in testing this took down the engine's worker connection and **rebooted the Machine**.
  Do not add libkrun workers to a Fly engine's `config.yaml`. To run sandboxed or
  image-based execution on Fly, run it as its own Fly Machine instead (the
  [Machines API](https://fly.io/docs/machines/) can launch one per job).
</Warning>

Everything else runs, verified on Fly:

- In-process workers (`iii-http`, `iii-state`, `iii-queue`, `iii-cron`, `iii-stream`,
  `configuration`, `iii-observability`).
- Registry workers that deploy as **binaries**: `iii worker add database` resolves to a
  binary (`x86_64-unknown-linux-gnu`) and runs as a plain host child process
  (`sandbox: n/a`), no `/dev/kvm` needed. Check a worker's deploy type before adding it;
  binary workers run, libkrun workers do not.
- Your own workers as separate Fly Machines over the 6PN.

## Verify the deployment

Once the engine app is live, the log shows each declared worker registering, including
any it auto-provisioned. Then call a route your worker registered through the public
domain:

```bash
fly logs -a iii-engine

curl https://iii-engine.fly.dev/orders -X POST \
  -H 'content-type: application/json' \
  -d '{ "sku": "abc", "qty": 1 }'
```

A response from your handler confirms the full path: Fly Proxy to `iii-http` to your
worker app over the 6PN to the `database` worker on the volume.

<Note>
  Redeploying the engine briefly re-registers HTTP routes, which can race a worker that
  still holds the old route. After an engine deploy, restart dependent worker apps
  (`fly apps restart <worker-app>`) so they reconnect and re-register.
</Note>

## Fly.io deployment checklist

This is the Fly-specific layer on top of the general
[Deployment checklist](./deployment#deployment-checklist).

**Image and build**

- [ ] `config.yaml` baked into the image (Fly has no build-time volume or file mount).
- [ ] Stateful or registry-worker deploys use the **root** debian base, not the
      non-root distroless image (volume write + `iii-worker`).
- [ ] Custom binary built for the Machine arch/libc (glibc, `amd64` unless you chose
      `arm64`).

**Networking**

- [ ] Engine binds `[::]`; workers dial `ws://iii-engine.internal:49134`.
- [ ] Only `3111` (and any required WS surface) is a public service; `49134` and `3112`
      stay private. Worker apps have no public IP (`fly ips release`).
- [ ] External worker access goes through WireGuard or a public RBAC listener, not the
      raw `49134` port.

**State and resiliency**

- [ ] No `DO NOT USE IN_MEMORY` warnings in the boot logs; `file_path`s live under
      `/data`.
- [ ] `fly volumes create` done in the engine's `primary_region`; volume snapshots/backups
      configured.
- [ ] `auto_stop_machines = "off"` and `min_machines_running = 1` on the stateful engine.
- [ ] `[[restart]] policy = "always"` on the engine, so a clean exit from a runtime
      config reload does not leave the Machine stopped (the default `on-failure` ignores
      code 0).
- [ ] Single engine Machine while on `file_based`; external adapters in place before
      scaling to multiple Machines or regions.

**Operations**

- [ ] TCP health check on `3111` (no in-container shell probes).
- [ ] Secrets set with `fly secrets`; CORS narrowed to real origins.
- [ ] Observability exporter set to `otlp` with a collector on the 6PN, or `memory`
      accepted for scope.
