## Settings live in a worker, not in config.yaml

iii splits configuration across two layers:

- **`config.yaml`** declares _which workers run_. See
  [Engine configuration](./engine#engine-configuration).
- **The `configuration` worker** owns _each worker's runtime settings_. It is a schema-validated,
  reactive registry that ships enabled by default: every worker registers its settings schema under
  its own entry (`iii-http`, `iii-state`, `iii-queue`, ...), and every change is validated against
  that schema and applied to the running worker â€” no engine restart.

The `config:` block under a worker in `config.yaml` is a **seed**: it is read once, on the first
boot after the worker registers its schema, to create the worker's configuration entry. From then on
the entry is the source of truth and the block is ignored. On the next boot the engine strips the
dead block from `config.yaml` and leaves a one-line breadcrumb comment pointing at where the value
now lives; the `- name:` line and everything else in the file are preserved as written.

<Note>
  This applies to workers that register a configuration schema â€” the built-in workers (`iii-http`,
  `iii-state`, `iii-queue`, `iii-cron`, `iii-stream`, `iii-pubsub`, `iii-observability`) and
  registry workers like `database` and `shell` all do. A worker that doesn't register one (for
  example `iii-worker-manager`) keeps reading its `config.yaml` block directly.
</Note>

## One file per worker

With the default `fs` adapter, every configuration entry is one YAML file named after the worker, in
`./config` at your project root:

```
config/
  iii-http.yaml
  iii-queue.yaml
  iii-state.yaml
  ...
```

Each file carries the entry's identity plus its current value (the registered JSON Schema is not
persisted â€” the worker re-registers it on every boot):

```yaml config/iii-http.yaml
id: iii-http
name: HTTP
description:
  HTTP server settings â€” host/port binding, CORS, request timeout, concurrency limit, and global
  middleware.
value:
  port: 3111
  host: 127.0.0.1
```

The directory is created on boot and watched from then on.

<Note>
  Before 0.21 the default directory was `./data/configuration`. When running on the default
  location, the engine performs a one-time migration on boot and moves any entries it finds there
  into `./config`. An explicit `directory:` override skips the migration.
</Note>

## Three ways to change a setting â€” all real-time

### Edit the file

Edit the fields under `value:` and save. The `fs` adapter watches the directory: the change is
validated against the worker's registered schema and applied through exactly the same path as
`configuration::set`. An edit that fails validation is rejected with a warning in the engine logs
and the previous good value stays in effect, so a typo can't take a worker down.

### Use the console

The [console worker](https://workers.iii.dev/workers/console)'s **Configuration** page lists every
registered entry and renders an editing form generated from its JSON Schema â€” typed fields,
per-adapter variants, and validation before save. `${VAR:default}` templates are shown and saved
verbatim, so env-driven values survive a console edit. Saving applies immediately.

```bash
iii worker add console
```

### Call `configuration::set`

`configuration::set` replaces the value for an entry, validates it against the registered schema,
and applies it â€” from the [CLI](./functions#trigger-functions-from-the-cli), from any SDK, or from
your own automation:

<Tabs>
  <Tab title="CLI">
    ```bash
    iii trigger configuration::get --json '{"id": "iii-http"}'
    iii trigger configuration::set --json '{"id": "iii-http", "value": {"port": 8080, "host": "127.0.0.1"}}'
    ```
  </Tab>
  <Tab title="Node / TypeScript">
    ```typescript
    await worker.trigger({
      function_id: "configuration::set",
      payload: { id: "iii-http", value: { port: 8080, host: "127.0.0.1" } },
    });
    ```
  </Tab>
</Tabs>

`configuration::get` reads one entry, `configuration::list` enumerates every entry (schemas only â€”
values are never included in the list), and `configuration::schema` returns the schema for one id.
See the [configuration worker docs](https://workers.iii.dev/workers/configuration) for the full
function reference and error codes.

## Environment variables in values

Values support the same `${VAR:default}` syntax as `config.yaml`. Templates are stored verbatim and
expanded against the live process environment on every read, so changing an env var propagates
without rewriting the stored value. A field that consists of a single placeholder is coerced to the
schema's scalar type after expansion â€” `port: ${HTTP_PORT:3111}` validates as the integer `3111`,
not a string. Pass `raw: true` to `configuration::get` to read the stored template form.

## How changes apply

Most settings hot-apply the moment they change: `iii-http` swaps CORS, timeout, and middleware
without dropping the listener, and binds a new `host`/`port` before releasing the old one. A few
fields are restart-tier â€” the change is recorded, logged, and takes effect at the next engine start
(for example `iii-state`'s storage adapter). Each worker's page on
[workers.iii.dev](https://workers.iii.dev/) documents its settings and how they apply.

## Changing where configuration lives

The configuration worker is configured like any other worker â€” through its own block in
`config.yaml`. To store the per-worker files somewhere else, set the `fs` adapter's `directory`:

```yaml config.yaml
workers:
  - name: configuration
    config:
      adapter:
        name: fs
        config:
          directory: ./config
```

The `bridge` adapter replaces local files entirely and delegates every `configuration::*` call to a
remote iii engine â€” useful when a fleet of engines should share one configuration source. An
optional `ttl_seconds` cleans up entries for ephemeral workers after their last subscriber
unregisters. See the [configuration worker docs](https://workers.iii.dev/workers/configuration) for
both.

## Reacting to configuration changes

Workers don't poll for changes â€” they bind a `configuration` trigger. The handler fires on every
register, set, and delete, including file edits and bridge-forwarded events from a remote engine:

```typescript
worker.registerFunction("link::on-config-change", async (event) => {
  console.log(`${event.id} changed:`, event.new_value);
  return {};
});

worker.registerTrigger({
  type: "configuration",
  function_id: "link::on-config-change",
  config: {
    configuration_id: "iii-http", // optional â€” omit to receive every id
    event_types: ["configuration:updated"], // optional â€” registered | updated | deleted
  },
});
```

This is the same mechanism the built-in workers use to hot-apply their own settings.
