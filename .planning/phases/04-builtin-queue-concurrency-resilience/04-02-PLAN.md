From engine/src/builtins/queue.rs (GroupedFifoWorker behavior):
```
- subscribe() checks concurrency > 1 for FIFO mode -> uses GroupedFifoWorker
- GroupedFifoWorker tracks active_groups, max_concurrent_groups = concurrency
- Pops jobs, checks group_id, skips if group already active
- Spawns tokio::spawn per group, removes group from active when done
- Skipped jobs are requeued via lpush
```

From engine/src/modules/queue/queue.rs (function queue consumer):
```
- Uses consume_function_queue which internally spawns a poll loop
- The poll loop calls move_delayed_to_waiting + pop per iteration
- FIFO behavior depends on the adapter's internal subscribe (for topic path)
- BUT for function queues, the builtin adapter polls and sends via mpsc channel
```

Critical insight from research: The function queue consumer path does NOT use GroupedFifoWorker.
It uses a simple poll loop in the adapter (engine/src/modules/queue/adapters/builtin/adapter.rs line 319).
The GroupedFifoWorker is used only in the direct subscribe path (topic-based).

**RESOLVED: The test MUST use Approach B (topic-based subscribe via enqueue_to_topic).**
This is the only path that reaches GroupedFifoWorker when concurrency > 1.
Follow the pattern in queue_integration.rs `full_roundtrip_fifo_preserves_order`.

**Test: fifo_multi_group_ordering_under_concurrent_load (QBLT-04)**

Follow the `full_roundtrip_fifo_preserves_order` pattern in `queue_integration.rs`:
1. Create engine with `Engine::new()` (with `ensure_default_meter()`)
2. Register handler function via `register_group_order_recording_function` with function_id matching the queue handler, group_field="group", seq_field="seq", processing_delay=Duration::from_millis(30)
3. Create `QueueCoreModule` with a custom FIFO config that has `concurrency >= 3` to enable GroupedFifoWorker:

```rust
let config = json!({
    "queue_configs": {
        "fifo-multi": {
            "type": "fifo",
            "message_group_field": "group",
            "concurrency": 3,
            "max_retries": 2,
            "backoff_ms": 100,
            "poll_interval_ms": 50
        }
    }
});
```

4. Call `module.initialize()` to start the subscribe-based consumer (this is where GroupedFifoWorker gets created)
5. Use `enqueue_to_topic` to send messages to the topic that maps to "fifo-multi" queue. Check how `full_roundtrip_fifo_preserves_order` determines its topic -- it uses the `enqueue` helper which goes through the function queue path. For the topic-based path, you need to determine the correct topic name. Read `engine/src/modules/queue/adapters/builtin/adapter.rs` subscribe method to find the topic format. It likely subscribes to a topic named after the queue (e.g., "fifo-multi").
6. Enqueue 3 groups x 5 messages each = 15 messages total. Each message: `{"group": "A"/"B"/"C", "seq": 0..4}`
7. Enqueue all messages for group A first, then B, then C (sequential enqueue is fine; the test verifies per-group ordering is preserved even when groups process concurrently)
8. Wait for all 15 messages to be processed. Use a polling loop: check `records.lock().await.len() == 15` with a timeout of 10 seconds (real wall-clock). Do NOT use start_paused since the system uses SystemTime.
9. Verify per-group FIFO ordering:
   ```rust
   let recs = records.lock().await;
   for group in ["A", "B", "C"] {
       let group_seqs: Vec<i64> = recs.iter()
           .filter(|(g, _)| g == group)
           .map(|(_, s)| *s)
           .collect();
       assert_eq!(group_seqs.len(), 5, "Group {group} should have 5 messages");
       for window in group_seqs.windows(2) {
           assert!(window[0] < window[1], "FIFO violation in group {group}: {} came before {}", window[0], window[1]);
       }
   }
   ```
10. Verify all 15 messages were processed (total records.len() == 15)

**Parallelism assertion (optional but recommended):**
The 30ms processing delay x 5 messages per group = 150ms if sequential within group. If groups run in parallel, total time should be ~150ms. If all 15 messages were sequential, it would take ~450ms. Record total elapsed time and assert it's < 400ms to prove some parallelism occurred. This is a soft assertion (log a warning instead of failing if timing is flaky in CI).

Use `#[tokio::test]` without start_paused. Use real wall-clock `std::time::Instant` for any timing measurements.
