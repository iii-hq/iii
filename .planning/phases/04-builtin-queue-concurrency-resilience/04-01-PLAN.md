**FIX-02: Atomic delayed-to-waiting transition (engine/src/builtins/queue.rs)**

In `move_delayed_to_waiting` method (line ~572), change the loop body from:

```rust
for job_id in &ready_jobs {
    self.kv_store.zrem(&delayed_key, job_id).await;
    self.kv_store.rpush(&waiting_key, job_id.clone()).await;
}
```

To:

```rust
for job_id in &ready_jobs {
    let removed = self.kv_store.zrem(&delayed_key, job_id).await;
    if removed {
        self.kv_store.rpush(&waiting_key, job_id.clone()).await;
    }
}
```

This ensures that only the caller that successfully removes the job from the delayed sorted set will push it to the waiting list. Concurrent callers that lose the race on `zrem` (returns false) skip the `rpush`.

**QBLT-06: Panic-safe function queue consumer (engine/src/modules/queue/queue.rs)**

In the inner `tokio::spawn` block (line ~317), wrap the `engine.call` + match block in `std::panic::AssertUnwindSafe` + `catch_unwind` to convert panics into nack calls. Replace the current spawned task body:

```rust
tokio::spawn(async move {
    let delivery_id = msg.delivery_id;
    let function_id = msg.function_id.clone();
    let attempt = msg.attempt;
    // ... span setup ...

    let result = std::panic::AssertUnwindSafe(async {
        engine.call(&function_id, msg.data).await
    })
    .catch_unwind()
    .instrument(span)
    .await;

    match result {
        Ok(Ok(_)) => {
            tracing::Span::current().record("otel.status_code", "OK");
            if let Err(e) = adapter.ack_function_queue(&queue_name, delivery_id).await {
                tracing::error!(error = %e, "Failed to ack message");
            }
        }
        Ok(Err(ref err)) => {
            tracing::Span::current().record("otel.status_code", "ERROR");
            tracing::warn!(
                function_id = %function_id,
                queue = %queue_name,
                attempt = %attempt,
                max_retries = %max_retries,
                error = ?err,
                "Function queue job failed"
            );
            if let Err(e) = adapter.nack_function_queue(&queue_name, delivery_id, attempt, max_retries).await {
                tracing::error!(error = %e, "Failed to nack message");
            }
        }
        Err(_panic) => {
            tracing::Span::current().record("otel.status_code", "ERROR");
            tracing::error!(
                function_id = %function_id,
                queue = %queue_name,
                attempt = %attempt,
                max_retries = %max_retries,
                "Function queue job panicked"
            );
            if let Err(e) = adapter.nack_function_queue(&queue_name, delivery_id, attempt, max_retries).await {
                tracing::error!(error = %e, "Failed to nack panicked message");
            }
        }
    }

    drop(permit);
});
```

This requires adding `use std::panic::AssertUnwindSafe;` and `use futures::FutureExt;` (for `.catch_unwind()` on futures) at the top of the file. Check if `futures` is already a dependency; if not, the `FutureExt` trait is re-exported by tokio or available via the `futures-util` crate. Alternatively, use `tokio::task::JoinHandle` to catch panics: instead of wrapping in catch_unwind, await the inner JoinHandle and check for `JoinError::is_panic()`. Choose whichever compiles cleanly with existing dependencies.

**Important:** The `catch_unwind` approach requires `AssertUnwindSafe` because `engine.call()` returns a Future that is not `UnwindSafe`. The `AssertUnwindSafe` wrapper is safe here because we handle the panic by nacking the message.

**Test 1: handler_panic_does_not_crash_worker (QBLT-06)**

This test proves that:
(a) The consumer loop continues after a handler panic
(b) The panicked message is nacked and follows the retry/DLQ path (thanks to our fix)
(c) Subsequent non-panicking messages are processed

Steps:
1. Create engine with `builtin_queue_config()` (standard queue, max_retries=2, backoff_ms=100)
2. Register `register_panicking_function` with function_id "test::panic_handler" and a success_count counter
3. Enqueue message 1 with `{"panic": true}` to "default" queue
4. Wait 500ms for the panicking message to process and be nacked
5. Enqueue message 2 with `{"panic": false}` to "default" queue
6. Wait 500ms for message 2 to process
7. Assert `success_count.load(Ordering::SeqCst) >= 1` (message 2 was processed)
8. Wait up to 3 seconds for the panicking message to exhaust retries (3 attempts x panic x 100ms backoff each)
9. Assert `dlq_count(engine, "default") >= 1` (panicked message eventually lands in DLQ after exhausting retries)

Use real wall-clock time (NOT start_paused) because the nack path uses SystemTime::now().
