**1. `register_payload_capturing_function`**
Registers a test function that stores every received `Value` payload into a shared `Arc<Mutex<Vec<Value>>>`. Pattern follows `register_order_recording_function` but captures the entire input, not a single field.

```rust
pub fn register_payload_capturing_function(
    engine: &Arc<Engine>,
    function_id: &str,
    captured: Arc<Mutex<Vec<Value>>>,
) {
    let function = Function {
        handler: Arc::new(move |_invocation_id, input| {
            let store = captured.clone();
            Box::pin(async move {
                store.lock().await.push(input);
                FunctionResult::Success(Some(json!({ "ok": true })))
            })
        }),
        _function_id: function_id.to_string(),
        _description: Some("payload capturing test handler".to_string()),
        request_format: None,
        response_format: None,
        metadata: None,
    };
    engine.functions.register_function(function_id.to_string(), function);
}
```

**2. `register_condition_function`**
Registers a condition function that returns `json!(true)` when `input[field] == expected_value`, and `json!(false)` otherwise. The `field` parameter is `&'static str` (same pattern as `register_order_recording_function`).

```rust
pub fn register_condition_function(
    engine: &Arc<Engine>,
    function_id: &str,
    field: &'static str,
    expected_value: Value,
) {
    let expected = expected_value.clone();
    let function = Function {
        handler: Arc::new(move |_invocation_id, input| {
            let exp = expected.clone();
            Box::pin(async move {
                let matches = input.get(field).map(|v| *v == exp).unwrap_or(false);
                FunctionResult::Success(Some(json!(matches)))
            })
        }),
        _function_id: function_id.to_string(),
        _description: Some("condition function".to_string()),
        request_format: None,
        response_format: None,
        metadata: None,
    };
    engine.functions.register_function(function_id.to_string(), function);
}
```

**3. `enqueue_to_topic`**
Calls `engine.call("enqueue", json!({"topic": topic, "data": data}))` and maps the result to `anyhow::Result<()>`.

```rust
pub async fn enqueue_to_topic(
    engine: &Engine,
    topic: &str,
    data: Value,
) -> anyhow::Result<()> {
    let result = engine.call("enqueue", json!({"topic": topic, "data": data})).await;
    match result {
        Ok(_) => Ok(()),
        Err(e) => Err(anyhow::anyhow!("enqueue to topic failed: {} - {}", e.code, e.message)),
    }
}
```

**Test 1: `enqueue_process_ack_preserves_payload` (QBLT-01)**

```rust
#[tokio::test]
async fn enqueue_process_ack_preserves_payload() {
    let engine = {
        iii::modules::observability::metrics::ensure_default_meter();
        Arc::new(Engine::new())
    };

    let captured_payloads: Arc<Mutex<Vec<Value>>> = Arc::new(Mutex::new(Vec::new()));
    register_payload_capturing_function(&engine, "test::payload_check", captured_payloads.clone());

    let module = QueueCoreModule::create(engine.clone(), Some(builtin_queue_config()))
        .await
        .expect("QueueCoreModule::create should succeed");
    module.initialize().await.expect("init should succeed");

    let sent_payload = json!({
        "order_id": 42,
        "items": [{"sku": "ABC-123", "qty": 3}],
        "metadata": {"source": "test", "timestamp": 1234567890},
        "tags": ["urgent", "vip"],
        "nested": {"a": {"b": {"c": true}}}
    });

    enqueue(&engine, "default", "test::payload_check", sent_payload.clone())
        .await
        .expect("enqueue should succeed");

    tokio::time::sleep(Duration::from_millis(500)).await;

    let received = captured_payloads.lock().await;
    assert_eq!(received.len(), 1, "handler should have been invoked exactly once");
    assert_eq!(
        received[0], sent_payload,
        "received payload must match sent payload structurally"
    );
}
```

Key details:
- Uses inline `Engine::new()` pattern (not `create_engine_with_queue`) because we need to register the handler BEFORE module initialization
- The payload is intentionally complex (nested objects, arrays, mixed types) to validate structural integrity
- Uses `assert_eq!` on `serde_json::Value` for structural equality comparison (field order irrelevant)
- 500ms sleep is sufficient for poll_interval_ms=50 builtin queue

**Test 2: `condition_based_filtering_routes_matching_messages_only` (QBLT-07)**
Uses the topic-based subscribe path with condition_function_id in trigger config.

```rust
#[tokio::test]
async fn condition_based_filtering_routes_matching_messages_only() {
    let engine = {
        iii::modules::observability::metrics::ensure_default_meter();
        Arc::new(Engine::new())
    };

    let call_count = Arc::new(AtomicU64::new(0));
    let captured_payloads: Arc<Mutex<Vec<Value>>> = Arc::new(Mutex::new(Vec::new()));

    // Register the handler that counts and captures payloads
    register_counting_function(&engine, "test::filtered_handler", call_count.clone());
    register_payload_capturing_function(&engine, "test::filtered_handler_capture", captured_payloads.clone());

    // Register condition function: only passes when category == "important"
    register_condition_function(&engine, "test::importance_condition", "category", json!("important"));

    let module = QueueCoreModule::create(engine.clone(), Some(builtin_queue_config()))
        .await
        .expect("QueueCoreModule::create should succeed");
    module.initialize().await.expect("init should succeed");

    let topic = format!("cond-test-{}", uuid::Uuid::new_v4());

    // Register trigger with condition_function_id AFTER initialize
    // (the "queue" trigger type is registered during initialize)
    engine
        .trigger_registry
        .register_trigger(Trigger {
            id: format!("trig-{}", uuid::Uuid::new_v4()),
            trigger_type: "queue".to_string(),
            function_id: "test::filtered_handler".to_string(),
            config: json!({
                "topic": &topic,
                "condition_function_id": "test::importance_condition",
                "queue_config": {
                    "max_retries": 2,
                    "backoff_ms": 100,
                    "poll_interval_ms": 50
                }
            }),
            worker_id: None,
        })
        .await
        .expect("register trigger should succeed");

    // Also register a second trigger WITHOUT condition on same topic to prove
    // the condition is what gates the first handler. This second handler captures
    // ALL payloads for comparison.
    engine
        .trigger_registry
        .register_trigger(Trigger {
            id: format!("trig-all-{}", uuid::Uuid::new_v4()),
            trigger_type: "queue".to_string(),
            function_id: "test::filtered_handler_capture".to_string(),
            config: json!({
                "topic": &topic,
                "queue_config": {
                    "max_retries": 2,
                    "backoff_ms": 100,
                    "poll_interval_ms": 50
                }
            }),
            worker_id: None,
        })
        .await
        .expect("register unconditional trigger should succeed");

    // Enqueue 3 messages via topic-based path: 2 non-matching, 1 matching
    enqueue_to_topic(&engine, &topic, json!({"category": "spam", "msg": "ignore me 1"}))
        .await
        .expect("enqueue spam 1 should succeed");

    enqueue_to_topic(&engine, &topic, json!({"category": "important", "msg": "process me"}))
        .await
        .expect("enqueue important should succeed");

    enqueue_to_topic(&engine, &topic, json!({"category": "spam", "msg": "ignore me 2"}))
        .await
        .expect("enqueue spam 2 should succeed");

    // Wait for all messages to be processed through both subscription paths
    tokio::time::sleep(Duration::from_millis(2000)).await;

    // The conditional handler should only be called for the "important" message
    let conditional_count = call_count.load(Ordering::SeqCst);
    assert_eq!(
        conditional_count, 1,
        "conditional handler should be invoked exactly once (only the 'important' message), got {conditional_count}"
    );

    // The unconditional handler should be called for ALL 3 messages
    let all_payloads = captured_payloads.lock().await;
    assert_eq!(
        all_payloads.len(), 3,
        "unconditional handler should receive all 3 messages, got {}",
        all_payloads.len()
    );
}
```

Key details:
- Uses inline `Engine::new()` (not `create_engine_with_queue`) to register handlers BEFORE module init
- UUID-prefixed topic name for parallel test safety
- Trigger registered AFTER `module.initialize()` (the "queue" trigger type is registered during initialize)
- Two triggers on same topic: one with condition (counting handler), one without (captures all). This proves the condition function is doing the filtering.
- `queue_config` in trigger config with matching `poll_interval_ms: 50` for predictable timing
- 2000ms sleep provides margin for 3 messages through topic subscribe path

IMPORTANT ANTI-PATTERN AVOIDANCE:
- QBLT-07 uses `enqueue_to_topic` (topic-based path), NOT `enqueue` (function queue path). The function queue path has NO condition checking.
- Condition function returns `json!(true)` / `json!(false)` (boolean), NOT string `"true"` / `"false"`.
- Each trigger has a unique UUID id to avoid id collision.

NOTE: If the second trigger approach (unconditional handler to verify all 3 messages arrive) causes issues because the builtin adapter might not support multiple subscribers on the same topic, simplify by removing the second trigger and only asserting that the conditional handler count is 1. The core assertion for QBLT-07 is that the conditional handler receives exactly 1 of 3 messages.
