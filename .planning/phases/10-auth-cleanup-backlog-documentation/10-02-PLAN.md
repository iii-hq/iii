**Section 1: Reliability Improvements (BLOG-01)**
Include ALL of these items from the Research inventory (each with Priority, Complexity, Category, Source cross-reference, Current State, Gap description, and Recommended Approach):

1. SEC-01: DNS rebinding SSRF mitigation via reqwest custom resolver to pin validated IPs
   - Priority: High, Complexity: Medium, Category: Security
   - Source: REQUIREMENTS.md SEC-01, PITFALLS.md Pitfall 3, FEATURES.md
   - Current state: URL validator checks IPs at validation time, but reqwest re-resolves at request time (TOCTOU gap)
   - Approach: Use reqwest `.resolve()` to pin validated IP or implement custom `reqwest::dns::Resolve`

2. SEC-02: Request/response body size limits to prevent resource exhaustion
   - Priority: High, Complexity: Low, Category: Security
   - Source: REQUIREMENTS.md SEC-02, FEATURES.md
   - Current state: body.clone() for HMAC signing has no size guard; response reads entire body into memory
   - Approach: Add configurable max_body_size check before serialization; limit response body reads

3. SEC-03: HMAC replay protection guidance and optional timestamp freshness enforcement
   - Priority: Low, Complexity: Low, Category: Security
   - Source: REQUIREMENTS.md SEC-03, PITFALLS.md Pitfall 6
   - Current state: HMAC signs timestamp:base64(body) but no receiver-side timestamp validation guidance
   - Approach: Document receiver-side timestamp validation; consider adding nonce to signed payload

4. REL-01: Circuit breaker for HTTP invocations to failing endpoints
   - Priority: Medium, Complexity: High, Category: Resilience
   - Source: REQUIREMENTS.md REL-01, FEATURES.md
   - Current state: No circuit breaker; failing endpoints receive unlimited retry traffic
   - Approach: Per-endpoint state tracking with threshold config, half-open probing

5. REL-02: Consumer crash recovery with automatic task restart
   - Priority: Medium, Complexity: Medium, Category: Resilience
   - Source: REQUIREMENTS.md REL-02, PITFALLS.md
   - Current state: If a spawned consumer task panics, the queue goes silent (handler panics are now caught per Phase 4 FIX, but the consumer task itself is not supervised)
   - Approach: Supervisor/restart logic for consumer tasks; detect and log consumer death

6. REL-03: Message deduplication with configurable retention window
   - Priority: Medium, Complexity: Medium, Category: Resilience
   - Source: REQUIREMENTS.md REL-03, FEATURES.md
   - Current state: message_id field exists on QueueMessage but no dedup logic
   - Approach: Idempotency window for recently-processed message_ids

7. REL-04: Redis adapter E2E test parity with builtin and RabbitMQ
   - Priority: Low, Complexity: Medium, Category: Testing
   - Source: REQUIREMENTS.md REL-04
   - Current state: Redis adapter exists but has no E2E test coverage; out of scope for current initiative
   - Approach: Mirror builtin/RabbitMQ E2E test patterns for Redis adapter

8. Auth env var cleanup beyond FIX-03 scope
   - Priority: Low, Complexity: Low, Category: Code Quality
   - Source: PITFALLS.md Pitfall 7, Research open question 3
   - Current state: http_functions/mod.rs and observability/mod.rs also use unsafe set_var (FIX-03 only covered auth.rs)
   - Approach: Apply same temp-env migration pattern to remaining test modules

**Section 2: Feature Additions (BLOG-02)**
Include ALL of these items (each with Priority, Complexity, Category, Source, Current State, Value Proposition, and Dependencies):

1. FEAT-01: Priority queues with configurable priority levels
   - Priority: Medium, Complexity: High, Category: Queue
   - Source: REQUIREMENTS.md FEAT-01, FEATURES.md
   - Current state: No priority queue support; all messages processed in FIFO/standard order
   - Value: Some messages are more urgent than others
   - Dependencies: New queue type, adapter changes across builtin/RabbitMQ

2. Scheduled/delayed message delivery user-facing API
   - Priority: Low, Complexity: Medium, Category: Queue
   - Source: FEATURES.md
   - Current state: Builtin has delayed-to-waiting mechanism internally but no user-facing API
   - Value: "Process this in 5 minutes" use case
   - Dependencies: API endpoint, SDK support

3. FEAT-02: DLQ redrive API with filtering for operational recovery
   - Priority: Medium, Complexity: Medium, Category: Queue
   - Source: REQUIREMENTS.md FEAT-02, FEATURES.md
   - Current state: redrive_dlq exists but moves ALL messages; no filtering
   - Value: Operators need to selectively redrive messages matching certain criteria
   - Dependencies: DLQ message inspection (done in Phase 3)

4. FEAT-03: HTTP invoker retry policies (configurable retry count, backoff strategy)
   - Priority: Medium, Complexity: High, Category: HTTP Invoker
   - Source: REQUIREMENTS.md FEAT-03, FEATURES.md
   - Current state: No automatic retry on HTTP invocation failure
   - Value: Transient failures (503, timeouts) should be retried automatically
   - Dependencies: Retry loop, backoff config, DLQ for failed invocations

5. Redis adapter hardening
   - Priority: Low, Complexity: Medium, Category: Adapter
   - Source: FEATURES.md, PROJECT.md Out of Scope
   - Current state: Redis adapter exists, not actively used, no E2E test coverage
   - Value: Users choosing Redis as queue backend get same reliability guarantees
   - Dependencies: REL-04 (E2E test parity first)

6. Queue depth metrics and alerting
   - Priority: Low, Complexity: Medium, Category: Observability
   - Source: FEATURES.md
   - Current state: OpenTelemetry metrics exist but queue depth is not instrumented
   - Value: Operators need to know when queues back up
   - Dependencies: OpenTelemetry metric integration

7. Message TTL / expiration
   - Priority: Low, Complexity: Medium, Category: Queue
   - Source: FEATURES.md
   - Current state: No per-message TTL support
   - Value: Stale messages should not be processed
   - Dependencies: Per-message metadata, adapter support

8. Webhook delivery retry with exponential backoff
   - Priority: Medium, Complexity: High, Category: HTTP Invoker
   - Source: FEATURES.md
   - Current state: Failed webhook delivery is not automatically retried
   - Value: Reliable webhook delivery is expected by consumers
   - Dependencies: Async retry loop, DLQ for webhooks, backoff config
