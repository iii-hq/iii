## Why iii? section

Create a **"Why iii?"** section directly below the hero. This section should explain the core thesis of iii in a clean, technical, high-signal way. It should feel like a systems product, not generic SaaS marketing.

### Intent
This section is where we explain what makes iii different. It should communicate that iii solves the problem of modern backend complexity into a minimal set of interoperable, real-time primitives. Do not make it feel fluffy, corporate, or feature-marketing-heavy.

### Section title
**Why iii?**

### Intro copy
Use a short paragraph like this in spirit:

> iii turns distributed backend complexity into a minimal set of real-time, interoperable primitives. Workers, functions, triggers, can connect, discover each other, and coordinate execution across languages as one system.

### Layout
Use a clean 2-column grid or card grid with exactly **6 items**. Each item should have:
- a short title
- a one-sentence explanation

Then, within the **same section but outside the grid**, add a separate emphasized line or mini-callout for:

**Capabilities become patterns**

This should feel like a concluding thesis statement for the section, not one of the core cards.

### Grid content

#### 1. Durable orchestration
Coordinate long-running, failure-tolerant execution across workers, and triggers.

#### 2. Interoperable execution
Execute across languages natively as if the system it were the one runtime.

#### 3. Live discovery
Functions and triggers exposed by one worker become visible and usable across the system in real time.

#### 4. Live extensibility
Add new workers and capabilities to a live iii system without redesigning the architecture.

#### 5. Live observability
Observe operations, traces, and system behavior across the entire connected stack in real time.

#### 6. Minimal primitives
Collapse distributed backend design into a minimal primitive surface area that humans and agents can reason about.

### Separate callout in the same section
After the grid, add a visually distinct statement such as:

**Capabilities become patterns**  
Agents, frameworks, tools, and higher-level abstractions can all be built on top of the same core system.

This callout should stand apart from the grid. It can be centered, larger, offset, boxed, or styled like a thesis statement.

### Design direction
- Match the current black, white, and yellow iii aesthetic
- Technical, minimal, sharp, monospaced, high contrast
- Should feel like infrastructure / systems software, not a startup landing page template
- Avoid long paragraphs
- Avoid icons unless they are extremely minimal
- Cards can have subtle borders, terminal-style structure, or industrial system-diagram energy
- The final "Capabilities become patterns" callout should feel important and memorable

### Avoid
Do not use vague phrases like:
- collaborate seamlessly across teams
- integrate without disruption
- perfect context for agents
- connect any service

Keep the language grounded in the actual iii thesis: orchestration, observability, extensibility, discoverability, interoperable execution, minimal primitives, and abstractions becoming patterns.
